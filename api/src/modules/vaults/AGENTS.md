# Vaults, contributions, flow rules, and shares

Spec: [docs/specs/api/0003-vaults-contributions-flows/index.md](../../../../docs/specs/api/0003-vaults-contributions-flows/index.md).

## File pointers

- `src/modules/vaults/` — `VaultsController`/`VaultsService` (create a `GROUP` vault, list/get, plus `createWithdrawableVault(tx, groupId, memberId)` called by the groups module, never exposed as an endpoint), `ContributionsController`/`ContributionsService` (record, list, reverse), `FlowRulesController`/`FlowRulesService` (create, list, replace, and the private distribution logic), `TransactionsService` (withdraw, reverse a single transaction, list, but no dedicated controller of its own), `SharesService` (recompute, list), `LedgerController` (`groups/:groupId/transactions`, `groups/:groupId/transactions/:id/reverse`, `groups/:groupId/me/withdraw`, `groups/:groupId/shares`; groups these together since none of them is one resource's CRUD), DTOs (`dto/`).
- `VaultsModule` exports `VaultsService`; `GroupsModule` imports it so `GroupsService`/`InvitesService` can call `createWithdrawableVault` (see `../groups/AGENTS.md`). `AppModule` also imports `VaultsModule` directly for its own controllers.

## Conventions

- **Money and percentage arithmetic always goes through `Prisma.Decimal`** (from `$prisma/client`), never plain JS numbers, to avoid floating point drift. Every distribution split (a flow rule's destinations, a `MEMBER_WITHDRAWABLE_VAULTS` split across members) computes each entry except the last by percentage, then assigns the last entry `total.minus(allocated)` so the parts always sum exactly to the whole, with rounding drift absorbed by the last one. `FlowDestination.sortOrder` (an int set at creation from the submitted array's index) is what "last" means; there is no other ordering signal on that table.
- **`FlowRulesService`'s distribution engine is direction generalized.** `applyContributionRules` (CONTRIBUTION, always CREDIT) and `applyProjectEntryRules` (`../projects/`'s `PROJECT_REVENUE`/`PROJECT_EXPENSE`, CREDIT or DEBIT respectively) share the same private `applyRule`/`splitAcrossMembers`/`writeVaultEntry` machinery, parameterized by a `direction`. A DEBIT write is balance guarded (`vault.updateMany` with `cachedBalance: { gte: amount }`), throwing 422 on insufficient funds rather than going negative; a CREDIT write is a plain `vault.update` with no such guard, matching contributions' existing behavior.
- **`Transaction.type` and `Transaction.sourceType` mean different things, do not conflate them.** `type` describes the mechanical shape of the entry (`INTERNAL_FLOW`, `WITHDRAWAL`, `CONTRIBUTION`, …); flow distributed transactions are always `type: 'INTERNAL_FLOW'`, never `type: 'CONTRIBUTION'`. `sourceType`/`sourceRefId` trace back to *why* the entry exists (`sourceType: 'CONTRIBUTION'`, `sourceRefId: <contribution id>`, or `sourceType: 'WITHDRAWAL'`). Any check for "is this transaction part of a contribution's distribution" must read `sourceType`, not `type`; `/debug` found and fixed exactly this bug in `TransactionsService.reverse()`.
- **A `CONTRIBUTION`-sourced transaction can only be reversed by reversing its parent `Contribution`.** `TransactionsService.reverse()` rejects (422) any transaction with `sourceType === 'CONTRIBUTION'`; only `ContributionsService.reverse()` may offset those, and it does so for every one of that contribution's transactions in one call.
- **A reversal transaction carries the same `sourceType`/`sourceRefId` as the transaction it reverses**, so `ContributionsService.reverse()`'s search for "this contribution's transactions still needing reversal" must filter on `reversedTransactionId: null` *and* `reversals: { none: {} }` together; filtering on `sourceType`/`sourceRefId` alone also matches the reversal rows themselves and double reverses. Also found and fixed by `/debug`.
- **The contribution create/reverse transactions use an explicit 15s Prisma timeout** (`{ timeout: 15_000 }`), not the 5s default: applying (or reversing) more than one active flow rule is several sequential per-destination writes inside one DB transaction, and the default budget isn't enough once a second rule is active. Raise this further (or batch/parallelize the writes) before adding a third simultaneous rule path.
- **Shares are recomputed, never historized.** `SharesService.recompute(tx, groupId)` upserts one `MemberShareCache` row per ACTIVE member (including a member with zero contributions, at 0%, not skipped) every time a contribution is recorded or reversed, inside the same transaction, before flow rules are applied so a `MEMBER_WITHDRAWABLE_VAULTS` destination always splits by the live post-change share.
- **Reuses `ListQueryDto`** from `../groups/dto/list-query.dto.ts` (imported via `$/groups/dto/list-query.dto`) rather than a second pagination shape; `ListTransactionsQueryDto` extends it with an optional `vaultId` filter.
- **Audit event codes**: `VAULT_CREATED`, `CONTRIBUTION_RECORDED`, `CONTRIBUTION_REVERSED`, `FLOW_RULE_CREATED`, `FLOW_RULE_REPLACED`, `WITHDRAWAL_DECLARED`, `TRANSACTION_REVERSED`, category `FINANCIAL`, seeded in `prisma/seed.ts` alongside the `MEMBER` category rows (see `../groups/AGENTS.md`); same free-string reference table pattern, no enum.
